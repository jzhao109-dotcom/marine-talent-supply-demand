```markdown
# Iterative Competency Clustering

## Core Rule

Do non-vector, evidence-preserving, iterative clustering. Do not compress thousands of raw abilities directly into a final dozens-level dictionary.

## Required Workflow

1. Audit input fields and evidence coverage.
2. Normalize text and exact-dedupe abilities without dropping source records.
3. Run one bounded clustering round.
4. Write `round_items.csv`, `round_dictionary.csv`, `review_flags.csv`, and `round_report.md`.
5. Review low-confidence and cross-domain flags before starting the next round.

## Round Targets

For policy-side data around 4427 unique abilities, prefer:

| Round | Target |
|---|---|
| 1 | 2500-3000 micro skills |
| 2 | 1000-1500 subcategory skills |
| 3 | 300-500 meso candidates |
| 4 | 40-80 final canonical skills |

## Required Tool

Use `scripts/iterative_competency_clustering.py` for CSV adaptation, exact dedupe, batch planning, resume files, and round output generation.

## Read When Needed

Read `references/competency-clustering-rules.md` before deciding whether two ability phrases can be merged.

## Red Flags

- One-step collapse from thousands of abilities to dozens of labels.
- Merging from `ability` alone without reading `evidence`.
- Dropping source files, chunk indexes, record IDs, or evidence.
- Forcing policy-side abilities into an existing supply-demand dictionary before independent clustering.
- Generic labels such as `管理能力`, `创新能力`, or `综合能力`.
```

- [ ] **Step 2: Write the reference merge rules**

Create `references\competency-clustering-rules.md` with these sections:

```markdown
# Competency Clustering Rules

## Merge Criteria

Merge abilities only when the professional object, core action, application setting, and evidence support the same mid-level skill.

## Keep Separate

Keep abilities separate when they differ by domain object, engineering phase, governance function, or required technical base.

## Label Rules

Use labels that name the domain object and capability module, ending with `能力`. Avoid slogans, job titles, isolated tools, or single-policy actions.

## Evidence Rules

Every cluster must retain representative evidence. If evidence is missing or too vague, mark the cluster as `review`.

## Policy-Side Examples

`海洋生态监测能力` and `海洋生态环境评价能力` may merge into `海洋生态监测评价能力` when evidence supports monitoring and evaluation tasks.

`海洋能资源调查能力` and `海洋能发电装备研发能力` should stay separate in early rounds because resource assessment and equipment R&D require different talent capabilities.

## Review Flags

Flag clusters that are too large, too generic, cross-domain, low-evidence, or created only to hit a target count.
```
